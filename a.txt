# DEFINITIVE PROMPT – AeroVend 1.0 (Visual Project Manager with Project Creation, Saving, and Loading)

**Act as a Principal Full‑Stack Software Engineer, Advanced Design System Designer, and Graphical Interface Architect with extensive experience in CAD environments, spatial diagramming (Figma/Miro), high‑density industrial HMI/SCADA systems, directed‑graph engines, and workflow automation (Trello/Butler).**

Build an interactive web application for **project management and visual presentation** that combines Trello (Kanban), Miro (infinite canvas + multi‑selection), SCADA (industrial monitoring), **directed‑graph diagrams** (Graph View), and **intelligent automation**. The application must allow **creating, saving, loading, and exporting projects**, whether starting from scratch or from a predefined template. The goal is to **explain a complex project to new people at a glance, with maximum clarity, control, and automatic reaction capability**, as well as to **build a project step by step from scratch**.

The pre‑loaded example project is **"AeroVend 1.0"**: a giant vending machine for fresh basil grown with high‑pressure aeroponics, a robotic harvesting arm, and an automated LED lighting system. The CEO **Leo** handles electronics, LEDs, robotic arm, and automation; **Christian** handles programming; **Ponce** handles fluid mechanics; **Sofía** handles basil, nutrients, and growth. Other unassigned areas appear as “Sin asignar”.

The aesthetic will be a **premium dark mode in the style of Vercel/Shadcn** (backgrounds `#0b0f19`, cards `#111827`, crisp white text, accents in blue, purple, and green), with an impeccable UI/UX layout where nothing feels cluttered. **FontAwesome** (or a similar icon library) will be used for interface icons, while emojis remain an option for the `icon` field of each card.

## 📁 1. PROJECT MANAGEMENT (NEW, SAVE, LOAD, EXPORT/IMPORT)

### 1.1 Default initial project
When the application opens for the first time, it automatically loads the **"AeroVend 1.0"** project from the JSON provided in the ANNEX. If the user already has a project saved in `localStorage`, that project will be loaded instead.

### 1.2 New project (from scratch)
- A **“New project”** button will be available in the top bar.
- On click, the user will be asked whether to **save the current project** before continuing (see 1.3). After confirming, a new empty project will be created with:
  - `projectName`: “Untitled project”
  - Default `canvasSettings` (canvas 8000×5000, zoom 1, no hidden zones, etc.)
  - `cards`: []
  - `connections`: []
- The user can then begin adding root cards using the **“+ Root Node”** button and freely connecting them.

### 1.3 Save project
- A **“Save”** button (floppy‑disk icon) will be available in the top bar.
- On click, a `.json` file with the complete state of the current project (equivalent to Export) will automatically download.
- **Additionally**, the project will be saved to `localStorage` under the key `currentProject`, so that the last saved state is restored when the page is reloaded (without needing to manually export). This action also updates the key `lastSavedProjectName` with the project’s name.
- If the project has never been saved and the user tries to create a new one or close the tab, a warning should be shown (can be implemented with `beforeunload`).

### 1.4 Load project
- An **“Open project”** button will be available in the top bar.
- It will allow selecting a `.json` file from the local file system and loading it, replacing the current state (with a warning if there are unsaved changes).
- A dropdown option will also allow **loading the example project** (the AeroVend 1.0 template), in case the user wants to return to the initial demo.

### 1.5 Export / Import
- The **Export JSON** (download current state) and **Import JSON** (load from file) buttons are kept as a complement to Save/Open.
- All states (including undo/redo history and personal notes) are cleared when creating a new project or loading one from a file.

## 🧩 3. MODULAR CARD ANATOMY (ZONES A–E) + ENHANCEMENTS

Each card is rendered with 5 strict visual zones, enriched with an icon, description, configurable image, adaptive internal scroll, compact view, and new intelligent indicators.

### Zone A – Main Header
- **Icon** (`icon`): emoji or CSS class, to the left of the title. If not defined, one is assigned automatically per area.
- **Title** with forced ellipsis (`white-space: nowrap; overflow: hidden; text-overflow: ellipsis`).
- **Description** (`description`): secondary text below the title (small font, color `#94a3b8`). In compact views it may be hidden or shown on hover. **Supports clickable links** in simple Markdown format (`[text](url)`) that open in a new tab.
- **Propagated alert indicator**: if the card (or any descendant) has `status: 'Bloqueado'` or `reqs` with `status: 'pending'`, a small red/orange flashing badge (using `pulse-border` animation) is displayed in the header, similar to an industrial alarm light.
- **Collapse button** (chevron) on the right.
- **“Quick view” toggle**: allows switching the card between detailed mode (zones A–E) and a **compact postcard‑like mode**, which shows the cover image (`image.src`) as background, the title, and the progress, hiding zones D and E. Ideal for rapid presentations or Focus mode. The state is stored per card (`compactView: boolean`).
- **The percentage must not be shown here.**

### Zone B – Progress Bar
- Styled native progress bar, 100% width.

### Zone C – Metadata and Local Control Row
- Avatar/name of `owner`.
- `area` label.
- **WBS code pill** (hierarchical, up to 10 levels) with subtle style (semi‑transparent background, small text).
- Explicit percentage text (e.g., “65%”).
- **Gear icon** (local configuration).
- **“IDEA” button** (lightbulb): opens a floating panel with a textarea for personal notes (`localStorage` key `userNotes_{cardId}`).
- **Mini pending blocks**: if `collapsed = true` and `showMiniPendingBlocks = true`, pills with the WBS of children with progress < 100% are injected, along with any child alerts (blockages, pending requirements).

### Zone D – Multi‑Dimensional Tab Panel
- Tabs: “Trabajo” (Checklist + `+ Tarea`), “Técnico”, “Económico”, “Reqs”.
- **Optional automatic flow rule (Auto‑complete)**:
  - In the “Trabajo” tab, a small toggle “Auto‑complete on finish” is shown next to the checklist.
  - If enabled, when the last pending task is checked as done, the card’s `status` automatically changes to `"Completado"`.
  - The toggle is stored in the card (`autoComplete: boolean`, default `false`).
- **Card polymorphism** (`cardType`): `"hito"`, `"dispositivo"`, `"subcontrata"`, `"estandar"`. Controls which tabs are displayed. If absent, `"estandar"` is assumed.

### Zone E – Dropzone Container for Sub‑blocks
- Space for child cards. Faint guide text when empty.
- **Image support** (`image`):
  - If `image.src` is not empty, it is rendered according to `image.position` (`"left"`, `"right"`, `"top"`).
  - Dimensions are configurable (`width`, `height` in px or `"auto"`). Use `object-fit: cover`.
  - **Optional auto‑adjustment**: when adding an image, the card may slightly expand its height (multiple of 5px) to avoid overflow, respecting the anchor point. This behavior can be disabled in the card’s configuration.
- **Adaptive internal scroll**: If a card is resized below the visible content, its interior must show **automatic vertical scroll** (with a minimalistic or hidden scrollbar, similar to a `no-scrollbar` class). This allows shrinking without losing access to information.

## 🖱️ 4. DRAG, MULTI‑SELECTION, RESIZE, AND KEYBOARD DELETE INTERACTIONS

### 4.1 Multi‑selection by drag (Marquee / Box Selection)
- Clicking on the background and dragging draws a semi‑transparent blue selection rectangle.
- On release, cards whose geometric centers intersect the rectangle become selected (highlighted border).
- **Batch actions**: move all together, change `owner`/`status`/`area`, **delete with Delete key** (see 4.4), collapse/expand.
- Exit selection with Escape or a click on empty space.

### 4.2 Individual drag
- Only from Zone A. 1:1 movement dividing deltas by `transform.scale`.
- **Ghost Box**: phantom silhouette at the future position/size.
- Snap on release: position → nearest multiple of 40px; size → nearest multiple of 5px.
- Collision: when dropping on another card, assign `parentId` and relative coordinates to the parent’s Zone E container. When dropping on the background, `parentId = null`.
- **Confirmation for destructive actions**: when deleting a card with children, or moving a card out of its parent, a confirmation dialog is shown.

### 4.3 Resize
- Bottom‑right handle. Ghost box with snap to 5px. Double‑click for Autofit (optimal height rounded to 5px).
- **ResizeObserver**: a `ResizeObserver` on each card will automatically redraw SVG connections and recalculate child layout when the card size changes for any reason (resize, autofit, collapse, image adjustment).

### 4.4 Keyboard delete (Suprimir)
- With one or more cards selected, pressing the **Delete** key will remove the selected cards after a confirmation dialog.
- There will be no visible delete button on the card to avoid cluttering the interface, but the keyboard action is fully functional (includes cleaning up connections, promoting orphan children or removing them based on user choice).

## 🧭 5. GLOBAL VISUALIZATION MODES (4 MAIN VIEWS) + PRESENTATION MODE

### 🗺️ Mode 1 – Spatial / Block Diagram (SCADA Canvas)
- Infinite canvas with GPU‑accelerated zoom/pan, background grid.
- All zones A–E visible according to configuration.
- SVG layer with animated Bézier pipeline connections between root cards.
- **Connection tooltips**: hovering over a pipe shows “Fin‑a‑Inicio: {from} → {to}” (using card titles).
- **Canvas minimap**: a small thumbnail in the bottom‑left corner showing the entire canvas with rectangles representing the cards. Clickable/draggable to move the camera.
- Optimizations: culling, `will-change: transform`, SVG isolation, debounce.
- **Quick filter by `owner`**: a selector in the top bar dims (reduces opacity) cards whose owner doesn’t match, to highlight one person’s cards during meetings.

### 📊 Mode 2 – Timeline / Gantt with Critical Path
- Bars proportional to duration, drag and resize to change dates.
- Real Finish‑to‑Start dependencies according to `connections`.
- **Critical Path calculation**: bars on the longest path are highlighted.
- **Delay propagation**: when a bar is moved, dependent cards shift accordingly.
- Virtualization, no Zone E.

### 📋 Mode 3 – Kanban Board
- Columns by `status`. Compact cards (icon, title, progress). Drag between columns to change `status`.
- Internal order by `startDate` or manual.
- No Zone E.

### 🕸️ Mode 4 – System Diagram (Graph View)
- Automatic layout (force‑directed or layered) of all root cards and children.
- Curved arrows between nodes according to `connections` and hierarchy.
- Zoom/pan, multi‑selection, owner filter.
- Each node shows icon, title, and progress; expandable on click.

### 🎥 Presentation Mode (Slide Show)
- “Present” button in the top bar.
- Automatically cycles through root cards (or selected ones) centering the camera one by one with smooth transitions (duration configurable, default 5s).
- Can be paused, advanced/rewound manually with arrow keys.
- Exit with Escape.

### ⭐ Camera Bookmarks
- In Canvas and Graph View, the user can save the current camera position + zoom with a name (e.g., “General View”, “Brazo Detail”).
- Bookmarks are stored in `localStorage` (within `canvasSettings`) and shown in a dropdown in the top bar to jump between them with one click.

## 📊 8. HIERARCHICAL DATA INTELLIGENCE (ROLLUP) + CASCADING ALERTS

### 8.1 Value rollup
- **Economic**: `presupuesto` and `gastado` of a parent = sum of all children (recursive). If the parent has its own value, it adds to the rollup.
- **Progress**: weighted average of direct children’s progress. If no children, manual value is kept.
- Calculated values are shown with a small “∑” icon and a distinct style (italic). If the user manually edits them, a warning is shown and the aggregation for that node is broken (the field becomes manual).

### 8.2 Cascading alert propagation (Upwards)
- **The system monitors** each card and its descendants for:
  - `status: 'Bloqueado'`
  - `reqs` with `status: 'pending'`
- If any card (at any depth) meets these conditions, its direct parent receives an incident counter, and so on up to the root.
- **Display**:
  - In collapsed cards, the alert indicator (red/orange flashing badge) in Zone A shows the total number of incidents under that subtree.
  - In Zone C, next to the mini pending blocks, active alerts are listed (e.g., “⚠️ 2 bloqueos, 1 req pendiente”).
- Thus, even with all cards collapsed, you know exactly where problems are.
