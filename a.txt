# DEFINITIVE PROMPT – VizPM v1.8 (Visual Project Manager OS)

**Act as a Principal Full‑Stack Software Engineer, Advanced Design System Designer, and Graphical Interface Architect with extensive experience in CAD environments, spatial diagramming (Figma/Miro), high‑density industrial HMI/SCADA systems, directed‑graph engines, and workflow automation (Trello/Butler).**

Build an interactive web application called **VizPM** (Visual Project Manager) for **project management and visual presentation** that combines Trello (Kanban), Miro (infinite canvas + multi‑selection), SCADA (industrial monitoring), **directed‑graph diagrams** (Graph View), and **intelligent automation**. The application must allow **creating, saving, loading, and versioning projects**, whether starting from scratch or from a predefined template. The goal is to **explain a complex project to new people at a glance, with maximum clarity, control, and automatic reaction capability**, as well as to **build a project step by step from scratch**.

**CRITICAL ARCHITECTURAL DISTINCTION:**
1. **The VizPM App Shell:** The global application frame. Contains a top global header (Logo, File menu, View toggles, Simple/Advanced mode, User profile) and a bottom taskbar (showing multiple open project windows).
2. **The Project Window:** A resizable, movable window *inside* the App Shell. Each open project is a window. Inside this window is the "Infinite Canvas", which is technically a giant root card named `main` containing the entire project hierarchy.

The pre‑loaded example project is **"AeroVend 1.0"**: a giant vending machine for fresh basil grown with high‑pressure aeroponics, a robotic harvesting arm, and an automated LED lighting system. The CEO **Leo** handles electronics, LEDs, robotic arm, and automation; **Christian** handles programming; **Ponce** handles fluid mechanics; **Sofía** handles basil, nutrients, and growth. Other unassigned areas appear as "Sin asignar".

The aesthetic will be a **premium dark mode in the style of Vercel/Shadcn** (backgrounds `#0b0f19`, cards `#111827`, crisp white text, accents in blue, purple, and green), with an impeccable UI/UX layout where nothing feels cluttered. **FontAwesome** (or a similar icon library) will be used for interface icons.

## 🖥️ 1. VIZPM APP SHELL (GLOBAL FRAME)

### 1.1 Top Global Header
- **"VizPM"** logo/text on the far left.
- **File Menu** dropdown: New project, Open project, Save project, Save version as draft, Load version, Settings.
- **View Mode selectors** (center): Canvas, Gantt, Kanban, Graph buttons.
- **Simple/Advanced toggle** (right): hides technical complexity for non‑technical users.
- **User avatar** (far right): shows current simulated user (Leo by default).
- **No standalone Export/Import buttons**; saving always downloads a JSON file, and opening loads a JSON file.

### 1.2 Bottom Taskbar (OS‑style)
- Shows icons of currently open project windows.
- Click to switch between projects, minimize, or close them.
- Each project window can be open simultaneously.

### 1.3 Global Persistence
- `localStorage` keys: `vizpm_global_settings`, `vizpm_open_windows`, `vizpm_projects` (array of project states), `vizpm_activity_log`.
- Auto‑save every 5 minutes + debounce on significant actions (drop, resize end, status change).

---

## 🪟 2. PROJECT WINDOW & INFINITE CANVAS (`main`)

### 2.1 Window Behavior (OS‑like)
- Each project opens as a **Window** with a title bar: `[Project Name] [—] [□] [X]`.
- **Barra de título superior**:
  - Left: project name (editable by clicking).
  - Center: view mode buttons (Canvas, Gantt, Kanban, Graph).
  - Right: minimize, maximize, close buttons.
- **Close (X)**: closes the window (returns to taskbar or shows save dialog).
- **Maximize**: occupies the full App Shell area.
- **Minimize**: collapses to taskbar icon.
- Windows can be **overlapped** (z‑index), **moved** by dragging the title bar, and **resized** independently.

### 2.2 The Canvas as Root Card (`main`)
- The infinite canvas is technically the root node with `id: "main"`.
- It has **all properties of a normal card**: title (project name), description, tabs, owner, area, progress, dates.
- Its `parentId` is `null` (absolute root).
- Contains all macro‑blocks as direct children.
- **Initial size**: 8000x5000 px (configurable by writing exact values in the side panel).

### 2.3 Agile Performance
- **Dragging**: uses a lightweight GPU‑accelerated "ghost proxy". Real DOM updates only on `drop`.
- **Resizing**: uses `transform: scale` and `will-change: transform`. `ResizeObserver` *only* recalculates SVG connection paths, never re‑renders the card DOM.
- **Marquee selection**: uses `requestAnimationFrame` and viewport culling.
- **Pan/Zoom**: Middle‑mouse drag or right‑click drag to pan. Scroll wheel to zoom (0.1x – 3x), centered on cursor.

## 🧩 3. SMART CARD HIERARCHY & ANATOMY

Cards have a strict **3‑level hierarchy**:
1. **Macro‑block** (direct child of `main`): connects to other macros via `connections`.
2. **Component** (inside macro): represents a subsystem.
3. **Sub‑component** (inside component): represents a specific part.

### 3.1 Clean UI (No Icons in Header)
- Minimalist dark mode.
- The `icon` field exists in JSON but is **never rendered** in the title bar.
- Title limited to **50 characters** (detailed) or **30 characters** (compact), truncated with ellipsis.

### 3.2 Smart Drop Zones
When dragging a card over another, 3 visual zones appear:
1. **Top edge**: insert as previous sibling (same parent).
2. **Center**: insert as child (nested inside).
3. **Bottom edge**: insert as next sibling (same parent).

### 3.3 Detailed Mode (`compactView: false`)
**Row 1 – Zone A: Title bar**
- Card title, left‑aligned, with `borderColor` as thin left accent stripe.
- Right end: collapse/expand chevron + "Quick view" toggle (eye icon).
- **No WBS code, no percentage, no icon.**

**Row 2 – Zone B: Progress bar**
- Thin, styled progress bar (full width), reflecting `progress` percentage.

**Row 3 – Zone C: Metadata & controls**
- Horizontal row:
  - **Owner avatars** (circle with initials, Google‑style). Click opens owner assignment modal.
  - **Area tag** (small pill with truncated area name).
  - **Percentage text** (e.g., "65%").
  - **Gear icon** (opens side panel for that card).
  - **"IDEA" lightbulb button** (opens personal note panel, saved in `localStorage`).
- **Mini pending blocks** (if collapsed and `showMiniPendingBlocks` is true): small pills with WBS codes of children with progress < 100%, plus alert indicators.

**Row 4 – Image display (if `image.src` is not empty)**
- Full‑width image according to `image.position`:
  - `"top"`: banner between Zone C and Zone D.
  - `"left"` / `"right"`: image becomes a column; remaining rows flow beside it.
- Image respects `width`/`height` and uses `object-fit: cover`.

**Row 5 – Zone D: Tab panel**
- Tabs: **"Trabajo"** (checklist + add task), **"Técnico"**, **"Económico"**, **"Reqs"**, and **"Comentarios"**.
- **Auto‑complete toggle**: In "Trabajo" tab, a switch "Auto‑completar al finalizar". When on and all tasks are checked, the card's `status` automatically becomes `"Completado"`. Stored in `autoComplete` field.

**Row 6 – Zone E: Sub‑block dropzone**
- Container for nested child cards. Faint guide text when empty.
- Hidden when card is collapsed.
- Child cards arranged according to parent's `layoutMode` (`"libre"`, `"fila"`, `"columna"`, `"grid"`).

**Adaptive internal scroll:** If a card is resized so that not all rows fit, rows 3‑6 get `overflow-y: auto` with a hidden scrollbar (class `no-scrollbar`), while rows 1‑2 remain fixed.

### 3.4 Compact Mode (`compactView: true`)
- **Postcard style**:
  - Full‑cover background from `image.src` (or dark gradient if no image).
  - Dark gradient overlay for text readability.
  - Overlaid content:
    - Top‑left: card **title** (truncated to 30 chars).
    - Bottom‑left: **owner avatars** (small) and **mini progress bar**.
  - Top‑right: **Quick view toggle** to go back.
- The card snaps to a fixed 4:3 aspect ratio; no other rows are visible.

## 📐 4. INTELLIGENT SIDE PANEL (Triangle Toggle)

### 4.1 Always‑Visible Triangle Button
- A small **triangle button (▶/◀)** is *always visible* on the right edge of the Project Window.
- **Auto‑expand**: clicking any card on the canvas automatically slides out this panel, displaying deep‑edit properties for that specific card.
- **Manual toggle**: clicking the triangle collapses/hides it.
- **Purpose**: keeps the canvas cards extremely clean. All editing happens in the panel.

### 4.2 Panel Content
- **Top section**: editable title + description + buttons (close panel, pin panel).
- **Middle section**: tabs with all properties (Trabajo, Técnico, Económico, Reqs, Comentarios).
- **Bottom section**: advanced configurations:
  - Notification rules (see section 10).
  - Permissions (future multi‑user).
  - Change history for that card.
  - Incoming/outgoing connections detailed.
  - Canvas size (only for `main` card).

## 📊 5. EXECUTIVE DASHBOARD (Collapsible Top Panel)

- A collapsible top panel *inside* the Project Window (below the window title bar).
- **Toggle button**: collapses/expands the dashboard.
- **KPIs in real‑time**:
  - **Global % Progress**: weighted average of all macro‑blocks.
  - **Budget vs Spent**: donut chart showing total `presupuesto` vs total `gastado`.
  - **Active Blockers**: list of cards with `status: 'Bloqueado'`.
  - **Critical Path**: cards on the critical path (from Gantt dependencies).
  - **Next 5 milestones**: upcoming `dueDate` values.
- **Height**: 80px when open, 0px when collapsed.
- Updates automatically when any card changes (reactive).

## 🔄 6. SINGLE SOURCE OF TRUTH (Reactive Views)

All views read from the exact same JSON state. Changes propagate instantly:

### 6.1 Mode 1 – Spatial / Block Diagram (SCADA Canvas)
- Infinite canvas with GPU transforms, background grid, SVG connections.
- All card rows visible as described in section 3.
- Connection tooltips, minimap, owner filter, culling, debounce.
- **Quick filter by owner**: dims non‑matching cards.

### 6.2 Mode 2 – Timeline / Gantt (fully functional)
- Root cards become horizontal bars on a calendar (days/weeks).
- Bar length = duration; drag/resize to change dates.
- Connections define Finish‑to‑Start dependencies → critical path highlighted.
- Delay propagation: moving a bar shifts dependents.
- Clicking a bar opens the side panel with full card details.

### 6.3 Mode 3 – Kanban (fully functional)
- Columns: "Sin Empezar", "En Progreso", "Bloqueado", "Completado".
- Compact Kanban cards (title, avatars, progress, due date).
- Drag between columns changes `status`.
- Order by `startDate` or manual.
- Click card → side panel opens.

### 6.4 Mode 4 – Graph View (fully functional)
- Automatic force‑directed/layered layout of all cards and connections.
- Arrows for hierarchy and `connections`.
- Nodes show title + progress; expandable.
- Zoom/pan, selection, owner filter.

### 6.5 Presentation Mode (Slide Show)
- "Present" button cycles through root cards (or selected ones), centering with smooth transitions (configurable duration, pause/advance with arrows, exit with Escape).

### 6.6 Camera Bookmarks
- Save named camera positions (in Canvas/Graph views), accessible from a dropdown.

## 📊 7. HIERARCHICAL DATA INTELLIGENCE (ROLLUP) + CASCADING ALERTS

### 7.1 Value Rollup
- **Economic**: `presupuesto` / `gastado` of a parent = sum of all descendants. If parent has manual value, it adds; editing manually breaks aggregation (warning shown).
- **Progress**: weighted average of direct children. No children → keep manual.
- Aggregated values shown with "∑" icon and italic style.

### 7.2 Cascading Alert Propagation
- Monitors each card's subtree for:
  - `status: 'Bloqueado'`
  - `reqs` with `status: 'pending'`
- Incidents bubble up to the root; a red blinking badge (CSS `pulse-border`) shows total count in Zone C (detailed) or as an overlay in compact mode.
- Even fully collapsed, you know there's a problem.

## 🔗 8. SVG CONNECTION LAYER WITH ResizeObserver

- Animated Bézier curves in Canvas and Graph views, with tooltips.
- In Gantt, connections define dependencies.
- `ResizeObserver` on every card redraws connections automatically on size change.

- Persisted in `localStorage`.

## 💬 10. COMMENTS ON CARDS

- Each card has a **"Comentarios" tab** in Zone D (alongside Trabajo, Técnico, etc.).
- This tab contains a public comment thread visible to all collaborators (future‑proof).
- Features:
  - Textarea to add a new comment (author = current user's name, default "Yo").
  - Each comment shows: author avatar (initials), timestamp, text.
  - Comments are stored in the card's JSON as an array: `comments: [{id, author, text, timestamp}]`.
  - The tab badge shows the number of comments.
- This is separate from personal "IDEA" notes (which remain private in localStorage).

## 🔔 11. SMART NOTIFICATIONS

### 11.1 Rules Engine
- Each card can have `notificationRules: []` array with rules like:
  ```json
  { "trigger": "status == 'Bloqueado'", "action": "badge + email" }
  { "trigger": "dueDate < 48h", "action": "badge + webhook" }
  { "trigger": "gastado > 90% * presupuesto", "action": "badge" }
  ```
- Rules are configured in the side panel (bottom section).

### 11.2 Actions
- **Internal badge**: red pulsing icon on the card.
- **Email** (mock API): sends notification (simulated).
- **Webhook** (mock Slack/Discord/Telegram): sends JSON payload (simulated).

### 11.3 Activity Feed / Change Log
- The application tracks all significant actions in an **activity log** stored in `localStorage` (key `activityLog`).
- Each entry: `{timestamp, actionType, cardId, cardTitle, description}`.
- The UI provides a **toggle button** in the top bar (icon: clock/history) that opens a **collapsible side panel** (from the right) showing the last 50 entries, sorted newest first.
- The panel is non‑intrusive, with a scrollable list; clicking an entry optionally highlights the relevant card.

- **Simple mode** (toggle in header):
  - Hides WBS codes, rollup math, graph view, technical tabs.

  - Everything visible: WBS, rollup, Gantt, Graph, all tabs.

## 💾 14. VERSION CONTROL / DRAFTS

- From the **File → Save version as draft** menu, the user can save a named snapshot of the entire project state.
- Snapshots are stored in `localStorage` under the key `projectVersions` as an array of `{name, timestamp, state}`.
- **File → Load version** shows a dropdown of saved drafts; selecting one restores that state (with confirmation).
- Auto‑save does not overwrite these named versions; they are explicit user actions.

## 👥 15. ROLES AND PERMISSIONS (STRUCTURE FOR FUTURE COLLABORATION)

- Although the current app is single‑user, the data model is **prepared for multi‑user collaboration**:
  - Project‑level field `users`: array of `{name, role}` where role is `"admin"`, `"write"`, or `"read"`.
  - Card‑level optional field `permissions`: can restrict which roles can edit specific cards.
- The UI does not enforce these rules yet, but the data structures exist for future integration.
- The current user can be simulated as "Leo" (admin) from the JSON.

## 💾 16. PERSISTENCE, COPY/PASTE, IDEA NOTES, AND DIALOGS

- `localStorage` keys:
  - `vizpm_projects` – array of all project states.
  - `vizpm_open_windows` – array of currently open window IDs.
  - `vizpm_global_settings` – global app settings.
  - `vizpm_activity_log` – array of change entries.
  - `undoStack`, `redoStack` – for undo/redo (per project).
  - `userNotes` – personal notes per card.
- Copy (Ctrl+C) on selected card copies its data; Ctrl+V pastes a duplicate with new ID and offset.
- IDEA button saves private notes per card (never exported).
- Confirmation dialogs: before creating new project without saving, deleting cards with children, moving a card out of its parent.
