# DEFINITIVE PROMPT – VizPM v1.7 (Visual Project Manager)

**Act as a Principal Full‑Stack Software Engineer, Advanced Design System Designer, and Graphical Interface Architect with extensive experience in CAD environments, spatial diagramming (Figma/Miro), high‑density industrial HMI/SCADA systems, directed‑graph engines, and workflow automation (Trello/Butler).**

Build an interactive web application called **VizPM** (Visual Project Manager) for **project management and visual presentation** that combines Trello (Kanban), Miro (infinite canvas + multi‑selection), SCADA (industrial monitoring), **directed‑graph diagrams** (Graph View), and **intelligent automation**. The application must allow **creating, saving, loading, and versioning projects**, whether starting from scratch or from a predefined template. The goal is to **explain a complex project to new people at a glance, with maximum clarity, control, and automatic reaction capability**, as well as to **build a project step by step from scratch**.

The application name **“VizPM”** will be displayed in the top‑left corner of the header, while the current project’s name will appear next to it as a subtitle or in a separate field. The pre‑loaded example project is **“AeroVend 1.0”**: a giant vending machine for fresh basil grown with high‑pressure aeroponics, a robotic harvesting arm, and an automated LED lighting system. The CEO **Leo** handles electronics, LEDs, robotic arm, and automation; **Christian** handles programming; **Ponce** handles fluid mechanics; **Sofía** handles basil, nutrients, and growth. Other unassigned areas appear as “Sin asignar”.

The aesthetic will be a **premium dark mode in the style of Vercel/Shadcn** (backgrounds `#0b0f19`, cards `#111827`, crisp white text, accents in blue, purple, and green), with an impeccable UI/UX layout where nothing feels cluttered. **FontAwesome** (or a similar icon library) will be used for interface icons.

## 📁 1. PROJECT MANAGEMENT (NEW, SAVE, LOAD, VERSIONING, AUTO‑SAVE)

### 1.1 Application header and File menu
The top bar contains a minimal set of controls:
- **“VizPM”** logo/text on the left.
- Next to it, the current **project name** (editable by clicking on it).
- A **“File” dropdown menu** with options: **New project**, **Open project**, **Save project**, **Save version as draft**, **Load version**.
- To the right, view mode selectors, undo/redo, search, and a toggle for the Activity Feed (see section 14).
- There are **no standalone Export/Import buttons**; saving always downloads a JSON file (see 1.3), and opening loads a JSON file (see 1.4).

### 1.2 Default initial project
When the application opens for the first time, it automatically loads the **“AeroVend 1.0”** project from the JSON provided in the ANNEX. If the user already has a project saved in `localStorage` (via auto‑save or manual save), that project will be loaded instead.

### 1.3 Save project
- **“Save”** from the File menu performs two actions:
  1. Downloads a `.json` file with the complete current state.
  2. Saves the project to `localStorage` under the key `currentProject` (for auto‑restore on reload). The project name is stored in `lastSavedProjectName`.
- If the project has never been saved and the user tries to create a new one or close the tab, a warning dialog appears (can be implemented with `beforeunload`).

### 1.4 Open project (Load)
- **“Open project”** from the File menu opens a file picker to load a `.json` file, replacing the current state (with a warning if there are unsaved changes).
- A sub‑option **“Load example project”** restores the original AeroVend 1.0 template.
- When loading, the undo/redo stacks and activity log are cleared (user can optionally keep personal notes).

### 1.5 Auto‑save
- **Auto‑save is enabled by default**. Every **5 minutes**, the application silently saves the current state to `localStorage` (key `currentProjectAutoSave`) without downloading a file. The interval can be changed in the settings (via a small gear icon in the File menu).
- Additionally, auto‑save triggers on significant actions (card drop, resize end, status change) after a 2‑second debounce to avoid excessive writes.

## 🧩 3. CLEAN CARD ANATOMY (WITHOUT DISPLAYING ICON) – TWO CLEAR DISPLAY MODES

Each card has two visual modes, toggled by the **“Quick view”** button in the header (toggle saved as `compactView` boolean). The card structure is a vertical stack of distinct rows. **The `icon` field is never rendered in the title bar**; the header remains minimal.

### 3.1 Detailed mode (default, `compactView: false`)
The card renders these rows from top to bottom:

**Row 1 – Zone A: Title bar**
- Only the card title, left‑aligned.
- Title limited to **50 characters**; longer text truncated with ellipsis.
- Background: dark, with the card’s `borderColor` as a thin left accent stripe.
- At the right end: **collapse/expand chevron** and **“Quick view” toggle button** (eye icon).
- **No WBS code, no percentage, no icon.**

**Row 2 – Zone B: Progress bar**
- Thin, styled progress bar (full width), reflecting `progress` percentage.

**Row 3 – Zone C: Metadata & controls**
- Horizontal row containing:
  - **Owner avatars** (circle with initials, Google‑style). If `owner` is a single name, show one circle with the first 3 letters; if multiple (comma‑separated), up to 3 circles and “+N” if more. Click opens owner assignment modal.
  - **Area tag** (small pill with truncated area name).
  - **Percentage text** (e.g., “65%”).
  - **Gear icon** (opens local configuration panel).
  - **“IDEA” lightbulb button** (opens personal note panel, saved in `localStorage`).
- **Mini pending blocks** (if card is collapsed and `showMiniPendingBlocks` is true): small pills with WBS codes of children with progress < 100%, plus any alert indicators.

**Row 4 – Image display (if `image.src` is not empty)**
- Full‑width image according to `image.position`:
  - `"top"`: banner between Zone C and Zone D.
  - `"left"` / `"right"`: the image becomes a column on the respective side; the remaining rows flow beside it.
- The image respects `width`/`height` and uses `object-fit: cover`.

**Row 5 – Zone D: Tab panel**
- Tabs: **“Trabajo”** (checklist + add task), **“Técnico”**, **“Económico”**, **“Reqs”**, and **“Comentarios”** (see section 12).
- **Auto‑complete toggle**: In “Trabajo” tab, a switch “Auto‑completar al finalizar”. When on and all tasks are checked, the card’s `status` automatically becomes `"Completado"`. Stored in `autoComplete` field.

**Row 6 – Zone E: Sub‑block dropzone**
- Container for nested child cards. Faint guide text when empty.
- Hidden when card is collapsed.
- Child cards arranged according to parent’s `layoutMode` (`"libre"`, `"fila"`, `"columna"`, `"grid"`).

**Adaptive internal scroll:** If a card is resized so that not all rows fit, rows 3‑6 get `overflow-y: auto` with a hidden scrollbar (class `no-scrollbar`), while rows 1‑2 remain fixed.

### 3.2 Compact mode (`compactView: true`)
- **Postcard style**:
  - Full‑cover background from `image.src` (or dark gradient if no image).
  - Dark gradient overlay for text readability.
  - Overlaid content:
    - Top‑left: card **title** (truncated to 30 chars).
    - Bottom‑left: **owner avatars** (small) and **mini progress bar**.
  - Top‑right: **Quick view toggle** to go back.
- The card snaps to a fixed 4:3 aspect ratio; no other rows are visible.

## 🖱️ 4. CANVAS PAN & ZOOM (EXCALIDRAW STYLE)

- **Middle‑mouse‑button drag** on canvas background pans the entire viewport (cursor changes to grabbing hand). This is the primary pan method.
- **Right‑click drag** also pans (secondary method).
- **Scroll wheel** zooms in/out, centered on cursor (zoom range 0.1x – 3x).
- All card‑level interactions (drag, resize, marquee) coexist without conflict.

---

## 🖱️ 5. INTERACTIONS: MULTI‑SELECTION, DRAG, RESIZE, DELETE

### 5.1 Multi‑selection (Marquee)
- Click on empty canvas and drag → semi‑transparent blue rectangle.
- On release, cards whose centers intersect the rectangle are selected (highlighted).
- Batch actions: move, change `owner`/`status`/`area` via right‑click menu, delete with **Suprimir** key (see 5.4), collapse/expand.
- Escape or click on empty space clears selection.

### 5.2 Individual card drag
- Only from Zone A (title bar). 1:1 movement dividing deltas by `transform.scale`.
- **Ghost Box**: phantom silhouette at the future snap position (multiple of 40px) and size (multiple of 5px).
- On drop: position snaps to 40px grid; size snaps to 5px if resizing.
- Collision detection: dropping on another card’s Zone E makes it a child (coordinates become relative). Dropping on canvas makes it root.
- Confirmation dialogs for destructive actions.

### 5.3 Resize
- Bottom‑right handle. Ghost box with snap to 5px. Double‑click handle → autofit (optimal height rounded to 5px).
- **ResizeObserver** on each card automatically redraws SVG connections and recalculates child layout whenever card size changes.

### 5.4 Keyboard delete
- With cards selected, pressing **Delete** removes them after confirmation. Connections cleaned, orphan children promoted or deleted per user choice.

## 🧭 6. FOUR FULLY FUNCTIONAL VIEWS + PRESENTATION MODE

**Every view must be fully developed** (not stubs). All interactive elements (drag, resize, tabs) work identically across views; only the graphical representation changes. The shared state is always the single project.

### 🗺️ Mode 1 – Spatial / Block Diagram (SCADA Canvas)
- Infinite canvas with GPU transforms, background grid, SVG connections.
- All card rows visible as described in section 3.
- Connection tooltips, minimap, owner filter, culling, debounce.
- **Quick filter by owner**: dims non‑matching cards.

### 📊 Mode 2 – Timeline / Gantt (fully functional)
- Root cards become horizontal bars on a calendar (days/weeks).
- Bar length = duration; drag/resize to change dates.
- Connections define Finish‑to‑Start dependencies → critical path highlighted.
- Delay propagation: moving a bar shifts dependents.
- Clicking a bar opens a floating modal with full card details (tabs, comments, etc.).

### 📋 Mode 3 – Kanban (fully functional)
- Columns: “Sin Empezar”, “En Progreso”, “Bloqueado”, “Completado”.
- Compact Kanban cards (title, avatars, progress, due date).
- Drag between columns changes `status`.
- Order by `startDate` or manual.
- Click card → full detail modal.

### 🕸️ Mode 4 – Graph View (fully functional)
- Automatic force‑directed/layered layout of all cards and connections.
- Arrows for hierarchy and `connections`.
- Nodes show title + progress; expandable.
- Zoom/pan, selection, owner filter.

### 🎥 Presentation Mode (Slide Show)
- “Present” button cycles through root cards (or selected ones), centering with smooth transitions (configurable duration, pause/advance with arrows, exit with Escape).

### ⭐ Camera Bookmarks
- Save named camera positions (in Canvas/Graph views), accessible from a dropdown.

## 📊 9. HIERARCHICAL DATA INTELLIGENCE (ROLLUP) + CASCADING ALERTS

### 9.1 Value rollup
- **Economic**: `presupuesto` / `gastado` of a parent = sum of all descendants. If parent has manual value, it adds; editing manually breaks aggregation (warning shown).
- **Progress**: weighted average of direct children. No children → keep manual.
- Aggregated values shown with “∑” icon and italic style.

### 9.2 Cascading alert propagation
- Monitors each card’s subtree for:
  - `status: 'Bloqueado'`
  - `reqs` with `status: 'pending'`
- Incidents bubble up to the root; a red blinking badge (CSS `pulse-border`) shows total count in Zone C (detailed) or as an overlay in compact mode.
- Even fully collapsed, you know there’s a problem.

- In Gantt, connections define dependencies.
- `ResizeObserver` on every card redraws connections automatically on size change.

## 💬 12. COMMENTS ON CARDS

- Each card has a **“Comentarios” tab** in Zone D (alongside Trabajo, Técnico, etc.).
- This tab contains a public comment thread visible to all collaborators (future‑proof).
- Features:
  - Textarea to add a new comment (author = current user’s name, default “Yo”).
  - Each comment shows: author avatar (initials), timestamp, text.
  - Comments are stored in the card’s JSON as an array: `comments: [{id, author, text, timestamp}]`.
  - The tab badge shows the number of comments.

## 🕐 13. ACTIVITY FEED / CHANGE LOG

- The application tracks all significant actions in an **activity log** stored in `localStorage` (key `activityLog`).
- Each entry: `{timestamp, actionType, cardId, cardTitle, description}`.
- The UI provides a **toggle button** in the top bar (icon: clock/history) that opens a **collapsible side panel** (from the right) showing the last 50 entries, sorted newest first.
- The panel is non‑intrusive, with a scrollable list; clicking an entry optionally highlights the relevant card.

## 💾 14. VERSION CONTROL / DRAFTS

- From the **File → Save version as draft** menu, the user can save a named snapshot of the entire project state.
- Snapshots are stored in `localStorage` under the key `projectVersions` as an array of `{name, timestamp, state}`.
- **File → Load version** shows a dropdown of saved drafts; selecting one restores that state (with confirmation).
- Auto‑save does not overwrite these named versions; they are explicit user actions.

## 👥 15. ROLES AND PERMISSIONS (STRUCTURE FOR FUTURE COLLABORATION)

- Although the current app is single‑user, the data model is **prepared for multi‑user collaboration**:
  - Project‑level field `users`: array of `{name, role}` where role is `"admin"`, `"write"`, or `"read"`.
  - Card‑level optional field `permissions`: can restrict which roles can edit specific cards.
- The UI does not enforce these rules yet, but the data structures exist for future integration.
- The current user can be simulated as “Leo” (admin) from the JSON.

## 💾 16. PERSISTENCE, COPY/PASTE, IDEA NOTES, AND DIALOGS

- `localStorage` keys:
  - `currentProject` – last saved state.

  - `projectVersions` – array of named drafts.
  - `activityLog` – array of change entries.
  - `undoStack`, `redoStack` – for undo/redo.
  - `userNotes` – personal notes per card.

- IDEA button saves private notes per card (never exported).
- Confirmation dialogs: before creating new project without saving, deleting cards with children, moving a card out of its parent.
